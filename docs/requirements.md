# Требования

Документ фиксирует текущие функциональные и нефункциональные требования к проекту, а также условия приёмки и готовности задач. Требования актуализируются по результатам командных ревью и согласований с заинтересованными сторонами.

## Контекст и цели
- Обеспечить единый источник актуальных требований для разработки, тестирования и релизного планирования.
- Согласовать критерии готовности задач к разработке (DoR) и завершённости результатов (DoD).
- Зафиксировать принятые в команде уточнения после ревью.

## Функциональные требования
- Предоставить основной лендинг с описанием продукта, ключевых преимуществ и сценариев использования.
- Реализовать форму обратной связи с валидацией обязательных полей и отправкой данных в бекенд API.
- Добавить блок часто задаваемых вопросов с возможностью сворачивания/разворачивания ответов.
- Поддержать мультиязычность интерфейса (минимум русский и английский) с переключением локали без перезагрузки страницы.
- Интегрировать аналитические события для основных пользовательских действий (просмотр секции, отправка формы, переключение языка).

## Нефункциональные требования
- Время загрузки первоначального контента не более 2 секунд на стабильном соединении 3G для целевых страниц.
- Оценка Lighthouse по Performance и Best Practices не ниже 90 для десктопной версии.
- Кодовая база покрыта линтингом и модульными тестами критичных компонентов (порог покрытия — договорной, но не ниже 70% для затронутых модулей).
- Все внешние зависимости зафиксированы в lock-файле и проходят регулярные обновления безопасности.

## Definition of Ready (DoR)
Задача считается готовой к разработке, если выполнены следующие пункты:
- Описаны бизнес-цель и ожидаемый результат для пользователя.
- Определены приёмочные критерии и метрики проверки (как минимум: UX-поток, валидационные правила, ожидаемые события аналитики).
- Проработаны макеты или UX-скицы, согласованные с дизайнером (при наличии визуальных изменений).
- Указаны зависимости от других задач/команд и наличие доступов к необходимым сервисам.
- Сформулирован план тестирования, включая особенности кросс-браузерной/кросс-девайсной проверки.

## Definition of Done (DoD)
Задача считается завершённой, если соблюдены условия:
- Реализованный функционал покрыт авто-тестами (юнит/компонентные, e2e — по договорённости) и проходит CI-пайплайн.
- Выполнен код-ревью без блокирующих замечаний, договорённые правки внесены.
- Обновлена документация (README, пользовательские инструкции, аналитические схемы) в соответствии с изменениями.
- Изменения задеплоены на тестовый/стейджинг стенд, результаты проверки задокументированы.
- Добавлены записи об аналитических событиях и обновлены дашборды/алерты, если требуется.

## Итоги командного ревью
- Проведено ревью требований и критериев DoR/DoD с командой разработчиков, тестирования и продукт-менеджером.
- По итогам ревью уточнены пороги по перформансу (Lighthouse ≥ 90) и обязательность аналитики для основных действий.
- Договорено пересматривать документ при каждом релизном цикле и фиксировать изменения в журнале актуализации (см. `docs/governance.md`).
